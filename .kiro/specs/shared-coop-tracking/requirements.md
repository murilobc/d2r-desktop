# Requirements Document

## Introduction

This feature adds a local network co-op session system to the D2R Tracker application. Multiple instances of the app running on the same local network can share a farming session in real-time, allowing groups of players to track combined statistics, items found, and run progress. The feature is entirely local (no internet required) and opt-in — solo functionality remains unchanged.

## Glossary

- **Host**: The D2R Tracker instance that creates and manages a co-op session, running the embedded WebSocket server
- **Guest**: A D2R Tracker instance that connects to a Host's session to share data
- **Session_Code**: A 6-character alphanumeric identifier generated by the Host for easy session sharing
- **Co-op_Session**: An active real-time shared tracking session between one Host and one or more Guests on the local network
- **WebSocket_Server**: An embedded TCP WebSocket server running in the Tauri Rust backend on the Host machine
- **Sync_Protocol**: The message format and rules governing real-time data exchange between Host and Guests
- **Player_Attribution**: Metadata associating each logged item or action with the specific player who performed it
- **Combined_Stats**: Aggregated statistics across all participants in a Co-op_Session (total runs, items, time)
- **Reconnection_Window**: The time period during which a disconnected Guest can rejoin without losing session context

## Requirements

### Requirement 1: Session Creation

**User Story:** As a host player, I want to create a co-op session with a shareable code, so that my friends on the local network can join my farming session.

#### Acceptance Criteria

1. WHEN the user activates the "Host Session" action, THE WebSocket_Server SHALL start listening on an available port on all local network interfaces
2. WHEN the WebSocket_Server starts successfully, THE Host SHALL generate a unique 6-character alphanumeric Session_Code
3. THE Session_Code SHALL be displayed in the Co-op Panel along with the Host's local IP address and port
4. WHILE a Co-op_Session is active, THE Host SHALL accept incoming WebSocket connections from Guests on the local network
5. IF the WebSocket_Server fails to bind to a port, THEN THE Host SHALL display an error message indicating the failure reason and remain in solo mode

### Requirement 2: Session Discovery and Joining

**User Story:** As a guest player, I want to join a co-op session by entering a session code or IP address, so that I can participate in a shared farming session.

#### Acceptance Criteria

1. WHEN the user enters a valid Session_Code and the Host IP address, THE Guest SHALL establish a WebSocket connection to the Host
2. WHEN the user enters the Host's IP address and port directly, THE Guest SHALL establish a WebSocket connection without requiring a Session_Code
3. WHEN a Guest successfully connects, THE Host SHALL send the current session state (run count, timer elapsed, item log, connected players) to the Guest
4. IF the Guest fails to connect within 5 seconds, THEN THE Guest SHALL display a connection error and allow the user to retry or cancel
5. IF the Session_Code provided by a Guest does not match the Host's Session_Code, THEN THE Host SHALL reject the connection with an invalid code error

### Requirement 3: Real-Time Session State Synchronization

**User Story:** As a co-op participant, I want to see live updates from all players in the session, so that I can track our combined farming progress.

#### Acceptance Criteria

1. WHILE a Co-op_Session is active, THE Sync_Protocol SHALL transmit session state changes within 100 milliseconds of occurrence
2. WHEN the Host starts a new run (split), THE WebSocket_Server SHALL broadcast the updated run count and timer reset to all connected Guests
3. WHEN a Guest logs an item, THE Guest SHALL send the item data with Player_Attribution to the Host
4. WHEN the Host receives a Guest item log, THE Host SHALL store the item in the Host's local database with Player_Attribution and broadcast the updated item log to all Guests
5. WHILE a Co-op_Session is active, THE Host SHALL broadcast the session timer value to all Guests every second

### Requirement 4: Combined Statistics Display

**User Story:** As a co-op participant, I want to see combined statistics across all players, so that I can measure our group's total farming efficiency.

#### Acceptance Criteria

1. WHILE a Co-op_Session is active, THE Combined_Stats SHALL display the total run count across all participants
2. WHILE a Co-op_Session is active, THE Combined_Stats SHALL display a unified item log from all participants
3. WHILE a Co-op_Session is active, THE Combined_Stats SHALL display the shared session timer elapsed time
4. THE Combined_Stats SHALL update in real-time as new runs are completed or items are logged by any participant

### Requirement 5: Per-Player Item Breakdown

**User Story:** As a co-op participant, I want to see which player found which items, so that I can track individual contributions.

#### Acceptance Criteria

1. WHEN an item is logged during a Co-op_Session, THE Sync_Protocol SHALL include the player name and profile identifier as Player_Attribution
2. THE Co-op Panel SHALL provide a per-player breakdown view showing items found by each participant
3. THE per-player breakdown SHALL display the player name, item count, and item list for each participant
4. WHILE a Co-op_Session is active, THE item log SHALL visually indicate which player found each item using the player name

### Requirement 6: Host Session Controls

**User Story:** As the host player, I want exclusive control over session timing (start, split, end), so that session pacing is coordinated by one player.

#### Acceptance Criteria

1. THE Host SHALL have exclusive ability to start, split (next run), pause, and end the Co-op_Session
2. WHEN the Host splits or ends a run, THE WebSocket_Server SHALL broadcast the action to all connected Guests
3. WHILE a Co-op_Session is active, THE Guest SHALL be able to log items but SHALL NOT be able to start, split, pause, or end the session
4. WHEN the Host ends the Co-op_Session, THE WebSocket_Server SHALL notify all Guests that the session has ended and close all connections

### Requirement 7: Guest Item Logging

**User Story:** As a guest player, I want to log items I find during the shared session, so that my drops contribute to the combined stats.

#### Acceptance Criteria

1. WHILE connected to a Co-op_Session, THE Guest SHALL present the item search interface for logging items
2. WHEN a Guest logs an item, THE Guest SHALL send the item data (name, type, rarity) to the Host via the Sync_Protocol
3. WHEN the Host receives an item from a Guest, THE Host SHALL store the item in the Host's database with the Guest's player name as Player_Attribution
4. IF the Guest loses connection before item delivery is confirmed, THEN THE Guest SHALL queue the item log and retry sending upon reconnection

### Requirement 8: Disconnection Handling

**User Story:** As a co-op participant, I want the system to handle network interruptions gracefully, so that temporary disconnections do not cause data loss.

#### Acceptance Criteria

1. IF a Guest loses connection to the Host, THEN THE Guest SHALL attempt to reconnect automatically every 3 seconds for a Reconnection_Window of 30 seconds
2. IF reconnection succeeds within the Reconnection_Window, THEN THE Host SHALL resend the current session state and the Guest SHALL resume participation without data loss
3. IF reconnection fails after the Reconnection_Window expires, THEN THE Guest SHALL fall back to solo mode and notify the user of the disconnection
4. WHEN a Guest disconnects, THE Host SHALL mark the Guest as disconnected in the connected players list but retain the Guest's session contributions
5. IF the Host application closes unexpectedly, THEN THE Guest SHALL detect the connection loss within 5 seconds and fall back to solo mode

### Requirement 9: Co-op Panel User Interface

**User Story:** As a user, I want a dedicated co-op panel in the application, so that I can easily host, join, and monitor co-op sessions.

#### Acceptance Criteria

1. THE Co-op Panel SHALL provide a "Host Session" button that starts the WebSocket_Server and displays the Session_Code
2. THE Co-op Panel SHALL provide a "Join Session" form with fields for IP address and Session_Code
3. WHILE a Co-op_Session is active, THE Co-op Panel SHALL display a list of connected players with their connection status (connected, disconnected)
4. WHILE a Co-op_Session is active, THE Co-op Panel SHALL display Combined_Stats (run count, items found, session time)
5. THE Co-op Panel SHALL be accessible from the sidebar navigation without disrupting the existing Run Tracker functionality

### Requirement 10: Solo Mode Compatibility

**User Story:** As a solo player, I want co-op features to be entirely opt-in, so that my existing workflow is not affected.

#### Acceptance Criteria

1. THE WebSocket_Server SHALL NOT start unless the user explicitly activates the "Host Session" action
2. THE application SHALL function identically to the current solo behavior when no Co-op_Session is active
3. WHEN a Co-op_Session ends (by Host or disconnection), THE application SHALL return to solo mode with all local data intact
4. THE Co-op_Session SHALL NOT modify the Guest's local database; only the Host's database stores session items with Player_Attribution
